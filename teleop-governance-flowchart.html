04 VENDOR API STACK — AIVISON SRC (Seer Robot Controller) — Not ROS2
RDSCore — Fleet Scheduler (HTTP REST + WebSocket)

Protocol

HTTP/1.1 REST on port 8088 · WebSocket on port 8089 for push telemetry at 10Hz

Key APIs

POST /setOrder · POST /terminate · POST /lock · POST /pause-navigation · POST /saveParams

Control Model

Order-based: task = sequence of blocks (waypoint + operation). Not real-time velocity. Used for navigation-level commands.

Safety Exposure

/saveParams can permanently change speed caps, geofence boundaries, navigation tolerances. No built-in access control.

Robokit — Direct Robot Control (Binary TCP)

Protocol

Custom binary frame (16-byte header: sync 0x5A + version + seq + length + type) over 6 TCP ports 19204–19301

Key APIs

API 2010 Open Loop Motion (raw vx/vy/w) · API 4005 Seize Control · API 6004 Soft E-stop · API 4101 Set Params Permanent

Control Model

Direct hardware control: raw velocity commands to motor driver. Real-time. Telemetry push at 100–200ms via port 19301.

Safety Exposure

API 2010 has NO geofencing, NO speed cap, NO collision check. It executes raw velocity immediately on the motor driver.

Critical Risk: RDSCore and Robokit can both control the same robot simultaneously. Neither is aware of what the other is doing. This is the primary architectural attack surface.
nventr wraps both APIs behind a unified safety-enforced abstraction
05 nventr ABSTRACTION LAYER — Wrapper, Safety Process & Adapter Architecture
Lock & Control Acquisition

Permanent Lock Hold

nventr holds API 4005 Seize Control on every robot at all times. SRC set to Control Mode via API 4003. Other controllers get rejection responses.

Lock Monitoring

Monitor locked field in telemetry push (19301). If locked=false appears → immediate re-lock + CRITICAL alert + soft E-stop.

Lock Release

Only during decommissioning or planned maintenance. Requires multi-party approval (Section 13, Rule 2).

Edge Safety Process

Speed Cap Enforcement

Clamp |vx|, |vy|, |w| to configurable maxima before calling API 2010. nventr's own config — does not rely on robot-side params.

Geofence Validation

On every command, query robot position from telemetry push. Validate target is within session's assigned zone. Reject + log if not.

Collision Pre-Check

Use block_x, block_y, block_reason from telemetry. If obstacle within threshold → rate-limit or halt motion.

E-Stop Authority

Only the Safety Process calls API 6004. No other nventr component has this permission. E-stop never routes through fleet scheduler.

Vendor Adapter Interface

Abstract Robot Contract

Each robot described by a capability contract: motion type, max speed, attachments, telemetry push interval. Vendor-independent.

Adapter Methods

VelocityCommand() · NavigateTo() · EmergencyStop() · GetTelemetry() · AcquireLock() · ReleaseLock()

Future-Proof

AIVISON adapter today calls Robokit/RDSCore. Future vendor adapter calls ROS2 or other protocol. Safety and governance layers never change.

Abstraction layer enforces the control chain below

07 HYBRID EDGE / CLOUD ARCHITECTURE
Edge Node (Per Warehouse)

What Runs Here

nventr Safety Process · Robokit lock holder · Watchdog heartbeat (50ms) · Vendor Adapter · Write-ahead audit log buffer · Hash-chain process · Local WORM append store

Why Edge

Safety enforcement must be <30ms latency. Network round-trip to cloud disqualifies cloud-only for real-time control.

Connectivity

LAN to robots <5ms · WAN to cloud for audit sync (tolerates higher latency, not in control path)

Failure Mode

If cloud sync fails, edge continues operating and buffering. Audit data queued, not lost. Sync resumes on reconnect.

Cloud (Central)

What Runs Here

WORM audit storage (L1–L5) · Hash chain verification · Risk scoring engine · Alert escalation pipeline · Geo-separated auth service · Compliance / auditor portals · Session recording archive

Why Cloud

Audit integrity, multi-facility correlation, geo-separated approval workflow, independent auditor access — all require a location not controlled by any single facility.

Geo-Separated Auth

Permanent parameter changes at one facility require approval from a person at a different facility. Cloud verifies approver location.

Every action at every step is captured in the audit pipeline
08 5-LAYER IMMUTABLE AUDIT — Full Reconstruction Capability

L1 · Command Log

Every input with operator ID, session, robot, task, timestamp, device. Append-only, hash-chained via async WAL pipeline. No deletes ever.

L2 · Environment

Robot telemetry from port 19301 push (100ms). Position, speed, battery, sensors, safety state. Both robot + edge timestamps stored.

L3 · Video + Screen

Robot camera feed, operator screen capture, control overlay, synchronized timestamps.

L4 · Communications

Voice, chat, radio, messaging — all timestamp-linked to active session.

L5 · Oversight Log

Overrides, E-stops (API 6004), alert dismissals, privilege changes, parameter modifications, session terminations.

Audit Pipeline: Edge WAL buffer → dedicated hash-chain process (SHA-256, async, never blocks control path) → local WORM append store → cloud sync every 30s or on session end → cloud WORM with daily anchor.
Stored in tamper-proof infrastructure
09 TAMPER RESISTANCE — No One Can Alter History

WORM Storage

Object store with retention lock — write once, read many. Edge + cloud.

Hash Chain

SHA-256 — each entry linked to previous. Store base files + configs that produce the hash (required for reproducibility).

Daily Anchor

Digest verified by internal team at a geo-separated facility — not an external third party.

Time Sync

NTP/PTP <5ms drift. Cross-check robot create_on timestamps against edge wall clock. Drift >5ms = alert.

Access Audit

All access to logs is itself logged and hash-chained.

Audit data feeds governance: roles, monitoring, and parameter control
10 SEPARATION OF DUTIES — Four Distinct Roles

Operator

CAN: Control assigned robots within task scope
CANNOT: Delete logs, modify thresholds, suppress alerts, self-certify

Site Manager

CAN: Assign operators, pause operations, request escalation
CANNOT: Delete audit trail, suppress safety alerts, silent override, self-review incidents

Compliance Officer

CAN: Review policies, verify controls are followed, suspend privileges, escalate to regional
CANNOT: Control any robot, alter operational parameters, access raw audit data

Auditor (Independent)

CAN: Read-only access to all logs, freeze artifacts, conduct independent reviews, produce audit reports
CANNOT: Control any robot, alter parameters, make operational decisions, modify audit data

12 PARAMETER MODIFICATION INTERCEPT

Intercept All Permanent Changes

nventr proxy blocks /saveParams (RDSCore) and API 4101 (Robokit). No direct calls allowed.

Multi-Party + Geo-Separated

Permanent param changes require approval from a person at a different facility. Cloud verifies approver location.

Before/After Hash

At session start: hash all robot params via API 1400. At session end: re-read and compare. Unapproved diff = CRITICAL alert.

15 ROBOT SAFETY BACKSTOPS — Enforced by nventr Edge Safety Process

Speed Caps

nventr clamps vx/vy/w before API 2010. Does not rely on robot-side params (can be changed via API 4101).

Geofence

Position checked against session zone on every command. nventr-enforced, not vendor-enforced.

Collision Avoidance

Uses block_x/y from telemetry push. Threshold-based halt. Never disabled.

E-Stop

API 6004 · Only Safety Process can call · Never routes through fleet scheduler · Highest priority

Heartbeat Watchdog

50ms heartbeat. On miss: API 6004 E-stop + API 3003 Cancel Nav in parallel. Does not wait for ControlMotionDuration.

Lock Guard

nventr holds API 4005 lock permanently. Lock-lost = CRITICAL alert + immediate E-stop + re-lock.

Any failure triggers forensic protocol
16 INCIDENT RESPONSE — ATC-Grade Forensic Protocol

1. Freeze

Auto-lock session logs, video archive, and telemetry

2. Package

Auto-generate replay: command log + 30–120s video + telemetry + screen + chat + hash

3. Assign

Independent reviewer — NOT the site manager who owns the operator

4. Investigate

Timeline reconstruction · Root cause: human error, malicious, software bug, or hardware

5. Remediate

System → patch · Human → retrain or suspend · Process → SOP update

6. Report

Timeline, evidence, actions, lessons learned — archived immutably

Design Principle: No single human can cause irreversible harm without detection. No single role can both act and erase evidence. Safety enforcement is built by nventr, not configured on the vendor — because the vendor APIs have no built-in safety.
