Command ACK & State Reconciliation

Protocol
sequenceDiagram participant T as Teleoperator participant C as Cloud Action Bus participant E as Edge Gateway participant R as Robot T->>C: Command (cmd_id, action, seq_no) C->>C: Validate RBAC + Policy C->>E: Forward Command via MQTT QoS 1 E-->>C: MQTT ACK received E->>R: Translate and Dispatch via MQTT R-->>E: Execution ACK (cmd_id, status, new_state) E-->>C: State Report (cmd_id, robot_state, ts) C->>C: Reconcile expected vs actual state C-->>T: Result Push via WebSocket Note over C,R: Timeout path - ACK missing after deadline C->>E: Retry or Query State E->>R: Health Probe R-->>E: Current State Snapshot E-->>C: State Reconciliation Report C->>C: Drift detected - Alert or Auto-correct
// ACK Envelope (robot → edge → cloud) { "cmd_id": "cmd-5a3f...", "seq_no": 42, "ack_status": "executed", // executed | rejected | partial | timeout "robot_state": { "status": "teleop", "position": { "x": 12.4, "y": 8.1 }, "battery": 72, "velocity": 0.3 }, "timestamp": "2026-03-04T10:42:17.003Z", "ntp_offset_ms": -2 } // State Reconciliation Logic (cloud-side) ON ack_received: IF ack.robot_state != expected_state → emit STATE_DRIFT alert IF ack.seq_no != expected_seq → emit SEQ_GAP, request replay IF ack timeout (>2s for critical) → retry x3, then escalate ON state_drift: log to audit → alert supervisor → optional auto-correct
A
ACK Model
Every command receives execution ACK with new robot state
R
Reconciliation
Cloud compares expected vs actual state on every ACK
T
Timeout
Critical commands: 2s ACK deadline, 3 retries
S
Seq Numbers
Per-robot monotonic sequence for gap detection
D
Drift Alert
State mismatch triggers audit log + supervisor alert
2W
Bidirectional
Commands down + ACK/state up — full closed loop

Fidelity Levels & Switching Rules

Levels
// Fidelity Level Definitions MINIMAL (idle) 1 fps keyframe-only ~0.1 Mbps No active session LOW (monitoring) 5 fps 480p ~0.5 Mbps Passive observation MEDIUM (standard teleop) 15 fps 720p ~2 Mbps Active teleop default HIGH (detailed) 30 fps 1080p ~5 Mbps Anomaly investigation MAX (full sensor) 30 fps 1080p+LiDAR+Audio ~15 Mbps Incident response (time-limited) // Upgrade Rules Upgrade MINIMAL → LOW auto: teleop session starts Upgrade LOW → MEDIUM auto: operator takes active control Upgrade MEDIUM → HIGH auto: anomaly engine flags event Upgrade HIGH → MAX manual: operator request + supervisor RBAC approval // Downgrade Rules Downgrade MAX → HIGH auto: MAX TTL expires (default 5 min) Downgrade HIGH → MEDIUM auto: anomaly resolved or 10 min timeout Downgrade MEDIUM → LOW auto: operator inactive 5 min Downgrade any → MINIMAL auto: teleop session ends Downgrade any → LOW force: battery <15% or bandwidth budget exceeded // Budget Example (10-robot site, 50 Mbps uplink) 1 robot at MAX = 15 Mbps 2 robots at HIGH = 10 Mbps 3 robots at MEDIUM = 6 Mbps 4 robots at LOW = 2 Mbps TOTAL = 33 Mbps of 50 Mbps budget ← 34% headroom
